## INNOV-1 — مرشّح ثبات طول العظم (Bone-Length Consistency Filter)

### الفكرة ولماذا تعمل
أطوال عظام الشخص ثابتة أثناء الجلسة، بينما أخطاء تقدير العمق (z) تشوّه الطول الظاهري لحظياً. مراقبة الطول = **كاشف أخطاء z مجاني** لا يحتاج ground truth: انحراف طول العضد 25% في إطار = لا تثق بزاوية الكوع في هذا الإطار. يعطي ما لا تعطيه visibility (التي قد تكون ممتازة بينما z خاطئ تماماً).

### الوصفة
1. **ملف جديد** `core/training-engine/.../geometry/BoneLengthMonitor.kt` (per-owner مثل `ElbowAngleEstimator` — يملكه مصدر الإطارات):
```kotlin
class BoneLengthMonitor {
    // أزواج العظام المرتبطة بالمفاصل العشرة: upperArm(11-13,12-14), forearm(13-15,14-16),
    // thigh(23-25,24-26), shin(25-27,26-28), torsoSide(11-23,12-24)
    private val medians = HashMap<Int, RunningMedian>() // key = boneId
    /** يعيد ثقة [0..1] لكل مفصل هذا الإطار، ويحدّث الوسيط الجاري. */
    fun assess(world: List<Landmark>): Map<String, Float>
}
```
   - `RunningMedian`: نافذة دائرية ~150 عينة (15 ثانية @10fps / 5 ثوانٍ @30) + وسيط تقريبي (P² algorithm أو نافذة مرتبة صغيرة — ‏150 عنصراً رخيصة).
   - ثقة العظم = `1 - clamp(|len - median| / (tolerance * median), 0, 1)` مع `tolerance = 0.20`؛ ثقة المفصل = min(ثقة عظمتيه).
   - Warm-up: أول ~30 إطاراً ثقة محايدة 1.0 (الوسيط غير ناضج).
2. **الدمج** في `PoseFrameAssembler.assemble`: معامل اختياري `boneConfidence: Map<String, Float>?` — الاستخدام الأول **قراءة فقط**: مرّره داخل `PoseFrame` (حقل جديد `jointMeasurementConfidence`) ليصل للمحرك والتقارير (INNOV-6). **لا تغيير سلوك في المرحلة الأولى.**
3. **المرحلة الثانية (بعد قياس)**: اربطه ببوابة 3D: ثقة < 0.4 تُعامل كرؤية منخفضة في `angleAt3D` (تغذي sticky) — عندها يصبح fallback per-joint مبنياً على إشارة حقيقية بدل visibility فقط (يكمل إصلاح F2).
4. **إعادة الضبط**: مع `resetAngleTracking()` الموجودة (تبديل عدسة/تمرين) — الوسيط يخص الشخص والجلسة.

### التحقق والقبول
- Unit: مدخل عالمي بأطوال ثابتة + إطار z-outlier واحد → ثقة المفصل تنهار في إطار الشذوذ فقط.
- جهاز (مع M-C): معدل الإطارات الموسومة منخفضة الثقة في جلسة سليمة < 5%؛ وفي سيناريو "ذراع نحو العدسة" يرتفع بوضوح حيث نعرف أن z يكذب.
- **قبول المرحلة 2**: تذبذب الزوايا (stddev على ثبات الوضع) ينخفض ≥15% بلا زيادة كمون ملحوظة.

## INNOV-2 — معايرة One-Euro لكل محور (Per-Axis Tuning: z أهدأ)

### الفكرة ولماذا تعمل
عمق MediaPipe (z) أضعف بنيوياً من x/y — نفس معاملات الترشيح للمحاور الثلاثة تعني إما ضجيج z يمر (beta يستجيب لسرعته الوهمية) أو x/y أبطأ من اللازم. ‏`minCutoff` أخفض وbeta أخفض لمحور z فقط = كتم اهتزاز العمق **دون** المساس باستجابة الحركة الحقيقية في مستوى الصورة.

### الوصفة
1. `OneEuroFilter3D` ([OneEuroFilter.kt:80](kmp-app/core/training-engine/src/commonMain/kotlin/com/movit/core/training/filter/OneEuroFilter.kt)) — وقّع جديد:
```kotlin
class OneEuroFilter3D(
    minCutoff: Float = 1.0f, beta: Float = 1.5f, dCutoff: Float = 1.0f,
    zMinCutoff: Float = minCutoff, zBeta: Float = beta,   // ← الافتراضي = السلوك الحالي حرفياً
) { private val zFilter = OneEuroFilter(zMinCutoff, zBeta, dCutoff) ... }
```
2. `PoseLandmarkSmoother` (الموحد للمنصتين) يمرر قيم z التجريبية من ثوابت مسماة:
```kotlin
object PoseSmootherTuning {
    const val Z_MIN_CUTOFF = 0.6f   // مرشح تجريبي — يقاس
    const val Z_BETA = 0.8f
}
```
3. طبّقها على **بنك world فقط** أولاً (`smoothWorld`) — بنك normalized يبقى كما هو (يغذي الرسم والفحوص 2D حيث لا z مؤثر أصلاً).

### التحقق والقبول
- ‏A/B على تسجيل ثابت (شخص ساكن في وضعية): stddev زاوية الركبة/الكوع 3D قبل/بعد — الهدف ≥20% انخفاض ضجيج.
- ‏A/B على حركة سريعة (rep سريع): زمن وصول الزاوية للقيمة القصوى لا يزيد > إطار واحد (لا نضيف كموناً في x/y — وهذا مضمون تصميمياً لأننا لم نلمسهما).
- تكافؤ المنصتين مضمون: الثوابت في common.

## INNOV-3 — الزاوية التنبؤية عند حدود الطور (Predictive Phase-Crossing)

### الفكرة ولماذا تعمل
سلسلة التنعيم (One-Euro → MA(3) → hysteresis 3°) تشتري استقراراً بثمن كمون ~200–300ms عند 10fps (يتقلص مع WP-19 لكنه يبقى محسوساً). ‏One-Euro يحسب مشتقة القيمة داخلياً أصلاً (`dx`) ويرميها. **التنبؤ خطوة واحدة للأمام** (`angle + velocity×dt`) عند اقتراب العتبة يعطي آلة الأطوار تأكيداً مبكراً بإطار أو اثنين — المستخدم يسمع العدّة عند قاع الحركة لا بعدها.

### الوصفة
1. اكشف السرعة من المرشح: `OneEuroFilter.lastFilteredVelocity: Float` (هي `filteredDx` المحسوبة أصلاً في `filter()` — سطر تخزين واحد).
2. مرّرها عبر السلسلة: `AngleSmoother.smooth` يعيد إلى جانب القيمة `velocityDegPerSec` (مشتقة MA بسيطة: فرق آخر قيمتين ÷ dt — أو الأدق: مرِّر سرعة One-Euro للمفصل عبر `PoseFrame` كحقل `angleVelocities: Map<String, Double>` يملؤه assembler من مرشحات world... **الأبسط والمعزول**: احسبها داخل `AngleSmoother` نفسه من قيمه — لا تغييرات في الالتقاط).
3. في `PhaseStateMachine.update`: عند مسافة ≤ `PREDICT_BAND = 6°` من عتبة الطور التالية وباتجاه الحركة الصحيح:
```kotlin
val predicted = angle + velocity * dtSec * PREDICT_HORIZON /* = 1 إطار */
val effective = if (sameDirection(velocity, threshold)) predicted else angle
```
   `effective` يُستخدم **فقط لقرار العبور**، لا يُسجَّل كقيمة زاوية (التقارير تبقى على الحقيقي).
4. حارس ضد الارتداد: التنبؤ لا يعمل إلا إذا كانت hysteresis ستُجتاز أيضاً بالقيمة المتنبأة (لا نضعف حماية الاهتزاز — نقدّمها زمنياً فقط).
5. علم داخلي `TimingPolicy.predictivePhaseCrossing: Boolean = false` → يُفعَّل بعد القياس.

### التحقق والقبول
- Unit: سلسلة زوايا اصطناعية بسرعة معلومة → العبور يحدث أبكر بإطار مقابل المسار الحالي، ولا عبور زائف عند تذبذب حول العتبة (اختبار jitter).
- جهاز: قياس "لحظة قاع الحركة الفيزيائية (فيديو مرجعي) → صوت العدّة" قبل/بعد — الهدف تحسّن ≥100ms عند 10fps و≥40ms عند 30fps، مع **صفر** فرق في إجمالي العدّات على جلسات الاختبار.

## INNOV-4 — إسقاط الزاوية على مستوى الحركة (Plane-Projected 3D Angles)

### الفكرة ولماذا تعمل
حركة المفصل المفرد (ركبة/كوع في تمرين نمطي) تحدث تقريباً في **مستوى ثابت**. معظم ضجيج z عمودي على هذا المستوى. إسقاط النقاط الثلاث على مستوى الحركة المقدَّر ثم حساب الزاوية = إلغاء المكوّن الأكثر ضجيجاً **مع الاحتفاظ بصحة 3D** (المستوى نفسه ثلاثي الأبعاد وقد يكون مائلاً في الفضاء). هذا حل عام أنظف من قواعد if/else الهجينة — ومرشح طبيعي ليحل محل `ElbowAngleEstimator` إن أثبتت M-C تفوقه.

### الوصفة
1. **ملف جديد** `geometry/MovementPlaneEstimator.kt` (per-joint، per-owner):
```kotlin
class MovementPlaneEstimator(private val windowSize: Int = 45) {
    // نافذة دائرية لمواضع (A,B,C) العالمية
    // كل N إطارات (مثلاً 10): PCA على النقاط المتمركزة → normal = eigenvector الأصغر
    // planeQuality = 1 - (λ3 / λ2)  ← قربها من 1 = الحركة مستوية فعلاً
    fun update(a: P3, b: P3, c: P3): PlaneEstimate?   // null أثناء warm-up
    fun project(p: P3, plane: PlaneEstimate): P3      // p - (p·n̂ - d)n̂
}
```
   - ‏PCA لمصفوفة 3×3: تنفيذ مغلق الصيغة (خصائص مصفوفة متماثلة 3×3 — ‏~40 سطراً، بلا مكتبات).
2. **الدمج التجريبي** (خلف علم بحثي، debug lab أولاً): في `angleAt3D` عند توفر `PlaneEstimate` بجودة ≥ 0.8: احسب الزاوية من النقاط المسقطة؛ وإلا المسار الحالي. اعرض في debug lab الثلاثي: 2D / 3D خام / plane-projected + جودة المستوى — **هذه أداة القياس لقرار M-C**.
3. **حالات يجب أن يتعامل معها**: تغيير الاتجاه بين الجولات (bilateral flip → reset)، الحركات غير المستوية فعلاً (planeQuality منخفضة → لا إسقاط — بهذا يفشل بأمان إلى السلوك الحالي)، وwarm-up أول rep.
4. **مسار الترقية إن نجح**: يستبدل استراتيجيات `MILD_DOWN/DEEP_DOWN` في الكوع أولاً (الأكثر حرجاً)، ثم يُعمَّم اختيارياً للمفاصل الأخرى — قرار لكل مفصل بالأرقام.

### التحقق والقبول (ضمن M-C)
- تسجيلات مرجعية بزوايا شبه معلومة (قرفصاء بعمق محدد أمام منقلة/فيديو جانبي): متوسط الخطأ المطلق مقابل المرجع — الهدف تحسّن ≥30% على السيناريو الأمامي (الأصعب) مقابل 3D الخام، وعدم تدهور على الجانبي.
- استقرار: stddev أثناء الثبات ≤ 3D الخام.

## INNOV-5 — فحوص الوضعية 3D بمحاذاة الجاذبية (Gravity-Aligned World Checks)

> هذه وصفة WP-20 التفصيلية — القرار والتسلسل في الخطة.

### الفكرة ولماذا تعمل
اليوم: فحوص الوضعية 2D على الصورة + تدوير landmarks بزاوية roll لتعويض ميل الجهاز — حل يعمل لكن: (أ) portrait-فقط بطبيعته، (ب) لا يعرف الأبعاد الحقيقية، (ج) يضيف تحويلاً لكل إطار. البديل المبدئي: عندنا **متجه الجاذبية الكامل** من نفس الحساس (‏TYPE_GRAVITY يعطي gx,gy,gz — نقرأ اليوم مكوّنتين ونرمي الثالثة!) وعندنا هيكل World بالمتر. دوران واحد يجعل «الأعلى» الحقيقي محوراً ثابتاً → كل فحوص العمودية/الوضعية/المسافة تصبح صحيحة **بأي ميل وأي اتجاه جهاز** وبعتبات فيزيائية (متر/درجة عن الجاذبية).

### الوصفة
1. **البوابة الحسية**: وسّع `AcquirableDeviceTiltPort` بـ `val gravity: Triple<Float,Float,Float>?` — ‏Android: القيم الثلاث من نفس `SensorEvent` (سطر واحد إضافي)؛ ‏iOS: ‏`CMDeviceMotion.gravity` جاهزة.
2. **التحويل**: `geometry/GravityAlignedSpace.kt`:
```kotlin
// worldY_target = -normalize(gravityInCameraSpace)
// ملاحظة محورية: World landmarks بإحداثيات محاذية للكاميرا — نفس إطار الحساس بعد
// تدوير الشاشة (طبّق displayRotation offset كما في T6 قبل البناء).
// R = rotation بين محور Y الحالي والمتجه المستهدف (Rodrigues حول المحور المتعامد)
fun alignToGravity(world: List<Landmark>, gravity: Vec3): List<Landmark>
```
3. **نموذج الفحص**: `PositionCheckType` يكتسب `space: CheckSpace = IMAGE_2D`؛ ‏`PositionValidator.validate` يبني `gravityWorld` مرة لكل إطار **فقط** إذا كان لدى الـ variant فحص GRAVITY_3D واحد على الأقل.
4. **معجم المشغّلات 3D**: عمودية الجذع = `angleBetween(shoulderMid−hipMid, +Y)`؛ ‏standing/lying = نفس المتجه ضد عتبة 45°؛ المسافة = `dist3D(a,b)` بالمتر مباشرة (عتبة config بالسنتيمتر!).
5. **fallback ثلاثي الطبقات**: لا gravity → المسار 2D المصحح بالميل (WP-16)؛ لا world → ‏2D؛ كلاهما متاح → 3D.
6. **الترحيل**: نوعاً نوعاً (posture → vertical → distance)، كل نوع باختبار fixture مائل 20° يثبت أن 3D ينجح حيث كان 2D غير المصحح يفشل.

### القبول
- جهاز على حامل مائل 25° + landscape: فحوص standing/lying وvertical تعطي نفس نتيجة الجهاز المستقيم.
- عتبة مسافة قدمين مكتوبة "40cm" في config تتحقق فيزيائياً (شريط قياس في QA).

## INNOV-6 — مقياس جودة القياس في تقرير الجلسة (Measurement Quality Telemetry)

### الفكرة ولماذا تعمل
كل القرارات أعلاه تحتاج عيوناً في الإنتاج. البنية موجودة أصلاً (`SessionQualityMeta` تُبنى لكل تمرين وتُرفع) — نضيف إليها **مقاييس القياس** فيصبح أي تدهور (fallback مفرط، fps هابط، عظام غير مستقرة) مرئياً في الداتا بلا أدوات جديدة، ويصير قرار "هل الهجين أفضل؟" و"هل 30fps ثابتة؟" قابلاً للإجابة من جلسات حقيقية.

### الوصفة
1. عدّادات رخيصة تتجمع أثناء الجلسة (في المحرك/مصدر الإطارات — كلها أعداد صحيحة، صفر تخصيص):
   - `frames3dPerJoint` / `frames2dFallbackPerJoint` (من sticky/angleAt3D)
   - `angleModeSwitchCount` (موجودة عملياً عبر `angleModeSwitchedJointCodes`)
   - `avgAchievedFps`، ‏`throughputProfileId`، ‏`adaptiveDowngrades` (من WP-19)
   - `boneConfidenceLowFrames` (من INNOV-1 عند تفعيله)
   - `tiltCorrectionActiveMs` + متوسط |roll| (يثبت عمل WP-16 في الميدان)
2. وسّع `SessionQualityMeta` بالحقول (nullable حتى لا تكسر أي مستهلك) ومرّرها في `resolveSessionQualityMeta` الموجودة في الـ VM.
3. لوحة/استعلام بسيط في الباك إند لاحقاً — خارج نطاق هذا الملف.

### القبول
حقول الجودة تظهر في تقرير post-training لجلسة حقيقية، وقيمها منطقية (3D ratio > 90% في إضاءة جيدة، fps ≈ الهدف).

## INNOV-7 — مزج 2D/3D موزون بالثقة (تعميم فلسفة الكوع) — بحثي

### الفكرة
مقدّر الكوع يجسد رؤية صحيحة: **لا مصدر واحد يُعتمد أعمى؛ الثقة تحدد المزيج**. تعميمها الرياضي بدل if/else: لكل مفصل `angle = w·ang3D + (1−w)·ang2D_aspectFixed` حيث `w = f(boneConfidence, planeQuality, facingRatio, |Δz|share)` — دالة لوجستية بسيطة بمعاملات قليلة تُعايَر من تسجيلات مرجعية (وليس MLP — قابلة للتفسير والضبط اليدوي).

### لماذا آخر القائمة
يحتاج مدخلات INNOV-1 (ثقة العظم) وINNOV-4 (جودة المستوى) وإصلاح F3 (‏ang2D سليم) أولاً، وقيمته الحدّية فوقها غير مثبتة — قد تكفي 1+4. يُقيَّم بعد M-C فقط إن بقيت فجوة دقة موثقة على سيناريوهات الواجهة الأمامية.

### وصفة مختصرة (إن فُتح)
- `JointAngleFusion.kt` يستقبل الإشارات الأربع → w لكل مفصل؛ معايرة offline بـ grid-search على مجموعة M-C المرجعية؛ يستبدل استراتيجيات الكوع فقط في البداية، ويُقارن ضده وضد INNOV-4 وجهاً لوجه.
