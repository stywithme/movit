# خطة تحويل النظام إلى نظام مرن وقابل للتوسع

> **المشروع**: POSE-2 — منصة تدريب ذكية  
> **التاريخ**: فبراير 2026  
> **الهدف**: تحويل النظام من بنية ثابتة (hardcoded) إلى نظام مرن بالكامل يمكن إدارته من لوحة التحكم

---

## الوضع الحالي — ملخص المشكلات

| العنصر | الوضع الحالي | المشكلة |
|--------|-------------|---------|
| المستويات | 5 مستويات ثابتة في الكود | لا يمكن تعديلها أو إضافة مستويات من الداشبورد |
| البرامج | API جاهزة لكن الداشبورد لا يدعم الحقول الجديدة | حقول الـ Prescription غير ظاهرة في واجهة إدارة البرامج |
| الاختبارات | اختبار واحد ثابت (Body Scan) بتمارين محددة في الكود | لا يمكن تعديل تمارين الاختبار أو إنشاء اختبارات مختلفة لكل مستوى |
| قواعد التقدم | 3 قواعد ثابتة في الـ Seed | لا يمكن للأدمن إدارة القواعد |

## المرحلة 1: المستويات الديناميكية

### 1.1 — الهدف
تحويل نظام المستويات من 5 مستويات ثابتة في الكود إلى نظام ديناميكي يقرأ من قاعدة البيانات، مع إمكانية الإضافة والحذف والتعديل من لوحة التحكم.

### 1.2 — التغييرات المطلوبة

#### Backend
- **إنشاء `LevelsAdminController`** — يوفر CRUD كامل للمستويات:
  - `GET /admin/levels` — عرض جميع المستويات مرتبة بالرقم
  - `POST /admin/levels` — إضافة مستوى جديد
  - `PUT /admin/levels/:id` — تعديل مستوى (الاسم، اللون، الوصف، الـ threshold، إعدادات التدريب الافتراضية)
  - `DELETE /admin/levels/:id` — حذف مستوى (مع حماية: لا يمكن حذف مستوى مرتبط بمستخدمين أو برامج)
  - `PUT /admin/levels/reorder` — إعادة ترتيب المستويات

- **تحويل `scoreToLevel()` من ثوابت إلى قراءة ديناميكية** — بدلاً من:
  ```
  if (score >= 85) return 5
  if (score >= 65) return 4
  ...
  ```
  يصبح:
  ```
  قراءة جميع المستويات من الـ DB ← ترتيب تنازلي بالـ entryThreshold ← أول مستوى score >= threshold
  ```
  مع **تخزين مؤقت (cache)** لتجنب استعلام DB في كل عملية حساب.

- **تحويل `LEVELS` array الثابت** إلى service يقرأ من DB مع cache مدته 5 دقائق.

- **إضافة حقل `maxThreshold`** اختياري لكل مستوى (حتى يكون النطاق واضحًا: من `entryThreshold` إلى `maxThreshold`).

- **حماية التكامل**:
  - عند حذف مستوى → التحقق من عدم وجود مستخدمين أو برامج مرتبطة
  - عند تعديل الـ thresholds → التحقق من عدم وجود تداخل بين النطاقات
  - منع حذف آخر مستوى (يجب وجود مستوى واحد على الأقل)

#### Admin Dashboard
- **صفحة إدارة المستويات** (`/admin/levels`):
  - جدول يعرض المستويات مرتبة بالرقم
  - كل صف يعرض: الرقم، الكود، الاسم (عربي/إنجليزي)، اللون، الـ threshold، عدد المستخدمين في هذا المستوى
  - أزرار: تعديل، حذف، إعادة ترتيب (drag & drop)
  
- **نموذج إنشاء/تعديل مستوى** — يشمل:
  - الرقم التسلسلي
  - الكود (مثل: `foundation`, `building`, ...)
  - الاسم (عربي + إنجليزي)
  - الوصف (عربي + إنجليزي)
  - اللون (color picker)
  - الأيقونة (اختياري)
  - عتبة الدخول (`entryThreshold`: النسبة المئوية 0-100)
  - إعدادات التدريب الافتراضية:
    - نطاق المجموعات (Sets: min-max)
    - نطاق التكرارات (Reps: min-max)
    - دليل الشدة (bodyweight_only / light / moderate / heavy / max_effort)
    - وقت الراحة الافتراضي بين المجموعات
    - مدة الجلسة (min-max بالدقائق)
    - تكرار أسبوعي (min-max أيام/أسبوع)

- **عرض بصري للنطاقات** — شريط ملون يعرض كل المستويات ونطاقاتها (0-100) حتى يتأكد الأدمن من عدم وجود فجوات أو تداخل.

#### Android
- **تحويل `FitnessLevel.fromBodyScore()`** ليقرأ الـ thresholds من الخادم بدلاً من ثوابت محلية.
- عند أول sync أو login، يتم تحميل قائمة المستويات وتخزينها محليًا.
- عند عدم الاتصال: استخدام آخر نسخة محفوظة.

## المرحلة 2: نظام الاختبارات المرن

### 2.1 — الهدف
تحويل نظام الاختبارات من اختبار واحد ثابت بتمارين محددة في الكود، إلى نظام يدعم:
- تعديل تمارين الاختبار الحالي
- إنشاء اختبارات مختلفة لكل مستوى
- ربط كل اختبار بمجموعة تمارين ومعايير تقييم

### 2.2 — المفاهيم الجديدة

#### Assessment Template (قالب الاختبار)
وحدة التعريف الأساسية للاختبار. تحتوي على:
- **الاسم** (عربي + إنجليزي)
- **الوصف**
- **النوع**: `initial` | `periodic` | `post_program` | `level_specific`
- **المستوى المستهدف** (اختياري) — يربط القالب بمستوى معين
- **نطاق المستويات** (min-max) — أي مستويات يظهر لها هذا الاختبار
- **الحالة**: مسودة / منشور
- **التمارين الأساسية** (Core Exercises) — تنفذ دائمًا
- **التمارين التكيفية** (Adaptive Exercises) — تنفذ بناءً على شروط
- **أوزان الحساب** (Domain Weights) — نسب حساب الـ Body Score:
  - وزن المرونة (Mobility Weight)
  - وزن التحكم (Control Weight)  
  - وزن التناظر (Symmetry Weight)
  - وزن السلامة (Safety Weight)

#### Assessment Exercise Entry (تمرين في الاختبار)
كل تمرين داخل القالب يحتوي على:
- **التمرين** (مرتبط بجدول Exercise الموجود)
- **الترتيب** في التسلسل
- **المنطقة المستهدفة** (hip, knee, shoulder, core, lower_back, balance)
- **الجانب** (left, right, center, bilateral)
- **إلزامي أم تكيفي** (core / adaptive)
- **شرط التفعيل** (للتمارين التكيفية فقط) — مثل:
  - "إذا ROM في overhead_squat < 60%"
  - "إذا stability في lunge < 50"
- **المعيار المرجعي** (Reference Norm) — القيمة الطبيعية للـ ROM بالدرجات
- **عتبات التقييم** — حدود الـ score لكل درجة (ممتاز/جيد/متوسط/ضعيف)

### 2.3 — التغييرات المطلوبة

#### Backend — Schema
- **إنشاء جدول `AssessmentTemplate`**:
  - `id`, `name`, `description`, `type`, `status`
  - `targetLevelId` (FK → Level, اختياري)
  - `levelRangeMin`, `levelRangeMax`
  - `domainWeights` (JSON: `{ mobility: 0.35, control: 0.25, symmetry: 0.20, safety: 0.20 }`)
  - `isDefault` (القالب الافتراضي عند عدم وجود قالب مخصص للمستوى)
  - `isPublished`
  - `sortOrder`

- **إنشاء جدول `AssessmentTemplateExercise`**:
  - `id`, `templateId` (FK → AssessmentTemplate)
  - `exerciseId` (FK → Exercise)
  - `sortOrder`
  - `targetRegion` (hip, knee, shoulder, etc.)
  - `side` (left, right, center, bilateral)
  - `entryType` (core / adaptive)
  - `activationCondition` (JSON، اختياري — للتمارين التكيفية)
  - `referenceNormDegrees` (Float — القيمة المرجعية للـ ROM)
  - `thresholds` (JSON: `{ excellent: 85, good: 65, average: 45, limited: 25 }`)

- **تعديل جدول `BodyScanResult`**:
  - إضافة `templateId` (FK → AssessmentTemplate, اختياري) — أي قالب استُخدم في هذا الاختبار

#### Backend — API
- **`AssessmentTemplateAdminController`** — CRUD كامل:
  - `GET /admin/assessment-templates` — قائمة القوالب
  - `POST /admin/assessment-templates` — إنشاء قالب
  - `GET /admin/assessment-templates/:id` — تفاصيل قالب (مع التمارين)
  - `PUT /admin/assessment-templates/:id` — تعديل قالب
  - `DELETE /admin/assessment-templates/:id` — حذف قالب
  - `POST /admin/assessment-templates/:id/publish` — نشر
  - `DELETE /admin/assessment-templates/:id/publish` — إلغاء نشر
  - `POST /admin/assessment-templates/:id/exercises` — إضافة تمرين
  - `PUT /admin/assessment-templates/:id/exercises/:exerciseId` — تعديل تمرين
  - `DELETE /admin/assessment-templates/:id/exercises/:exerciseId` — حذف تمرين
  - `PUT /admin/assessment-templates/:id/exercises/reorder` — إعادة ترتيب

- **Mobile endpoint**:
  - `GET /mobile/assessment-templates/resolve` — يحدد القالب المناسب بناءً على مستوى المستخدم:
    1. يبحث عن قالب مرتبط بالمستوى الحالي
    2. لو ما لقى → يبحث عن قالب ضمن نطاق المستوى
    3. لو ما لقى → يستخدم القالب الافتراضي (`isDefault = true`)

#### Admin Dashboard
- **صفحة إدارة قوالب الاختبارات** (`/admin/assessment-templates`):
  - جدول يعرض القوالب: الاسم، النوع، المستوى المرتبط، عدد التمارين، الحالة
  - فلترة بالنوع والمستوى والحالة

- **صفحة إنشاء/تعديل قالب** (`/admin/assessment-templates/[id]/edit`):
  - **قسم المعلومات الأساسية**:
    - الاسم (عربي + إنجليزي)
    - الوصف
    - النوع (initial / periodic / post_program / level_specific)
    - المستوى المستهدف (dropdown من المستويات الموجودة)
    - نطاق المستويات (min-max sliders)
    - قالب افتراضي؟ (toggle)
  
  - **قسم أوزان الحساب** (Domain Weights):
    - 4 sliders (Mobility, Control, Symmetry, Safety) مجموعها = 100%
    - عرض بصري (pie chart أو bar) يتحدث مباشرة
  
  - **قسم التمارين**:
    - قائمة التمارين الأساسية (Core) — ترتيب بالسحب والإفلات
    - قائمة التمارين التكيفية (Adaptive) — مع شرط التفعيل
    - زر "إضافة تمرين" → يفتح dialog لاختيار التمرين من قائمة التمارين المنشورة
    - لكل تمرين:
      - المنطقة المستهدفة (dropdown)
      - الجانب (dropdown)
      - المعيار المرجعي (ROM بالدرجات)
      - عتبات التقييم (4 حقول رقمية)
      - شرط التفعيل (للتمارين التكيفية — JSON editor بسيط أو rule builder)

- **عرض بصري لتغطية المناطق** — رسم بياني يوضح أي مناطق من الجسم مغطاة بالتمارين وأيها ناقصة.

#### Android
- **تعديل `AssessmentSessionActivity`**:
  - بدلاً من قراءة `coreExercises` من ثوابت محلية → يطلب القالب من الخادم عبر `/mobile/assessment-templates/resolve`
  - يحمل قائمة التمارين ومعلوماتها من القالب
  - يحفظ `templateId` مع نتيجة الاختبار عند الرفع

- **تعديل `AssessmentEngine`**:
  - بدلاً من `extractRegions()` hardcoded → يقرأ mapping المناطق من القالب
  - بدلاً من أوزان Body Score الثابتة (35%, 25%, 20%, 20%) → يقرأها من `domainWeights` في القالب
  - بدلاً من `ReferenceNormsProvider` الثابت → يقرأ المعايير المرجعية من القالب

- **تعديل `AdaptiveBatteryManager`**:
  - يقرأ شروط التفعيل من القالب بدلاً من المنطق المكتوب في الكود

- **Offline fallback**: تخزين آخر قالب محمل محليًا للاستخدام عند عدم الاتصال.

## المرحلة 3: تطوير إدارة البرامج في الداشبورد

### 3.1 — الهدف
تحديث واجهة إدارة البرامج في الداشبورد لتدعم جميع الحقول الجديدة (Prescription metadata) مع تجربة مستخدم سلسة.

### 3.2 — التغييرات المطلوبة في الداشبورد

#### تحديث صفحة إنشاء/تعديل البرنامج
الصفحة الحالية تدعم: الاسم، الوصف، المدة، الصعوبة، الغلاف، التاجات، والأسابيع/الأيام/الجلسات.

**إضافة قسم جديد: "إعدادات الوصفة الذكية" (Prescription Settings)**:

- **نوع البرنامج** (`type`):
  - Radio buttons: تدريب عام | مرونة وحركة | علاجي/تصحيحي
  - مع وصف قصير لكل نوع

- **المجال المستهدف** (`targetDomain`):
  - Dropdown: غير محدد | المرونة (Mobility) | القوة (Strength) | التحكم (Control) | التناظر (Symmetry)
  - يُفعّل فقط عندما البرنامج ليس "تدريب عام"

- **المناطق المستهدفة** (`targetRegions`):
  - Multi-select chips: الكتف | الورك | العمود الفقري | الركبة | الأساسي | التوازن
  - اختياري — لتحديد مناطق الجسم التي يركز عليها البرنامج

- **نطاق المستويات** (`levelRangeMin` - `levelRangeMax`):
  - Range slider مزدوج يعرض المستويات بألوانها
  - يعتمد على المستويات الموجودة في DB (ديناميكي)
  - مثال: "من المستوى 2 (بناء) إلى المستوى 4 (متقدم)"

- **شروط الدخول** (`entryCriteria`):
  - Rule builder بسيط — يسمح بتحديد شروط مثل:
    - Body Score ≥ X أو ≤ X
    - Mobility Score ≥ X
    - أي domain score مع أي عامل مقارنة
  - يُخزن كـ JSON

- **شروط الخروج/الانتقال** (`exitCriteria`):
  - نفس الـ rule builder
  - يحدد متى المتدرب "تخطى" هذا البرنامج

- **موانع الاستخدام** (`contraindications`):
  - Multi-select من المناطق — يمنع وصف البرنامج للمتدربين مع safety gates في هذه المناطق

- **أولوية الوصف** (`prescriptionPriority`):
  - Slider رقمي (1-100) — كلما كان أقل كان أولوية الوصف أعلى
  - مع شرح: "رقم أقل = يظهر أولاً في التوصيات"

- **سلسلة البرامج** (Program Chain):
  - **البرنامج المطلوب مسبقًا** (`prerequisiteProgramId`): Dropdown من البرامج المنشورة
  - **البرنامج التالي** (`nextProgramId`): Dropdown من البرامج المنشورة
  - عرض بصري للسلسلة: `برنامج أساسي → هذا البرنامج → برنامج متقدم`

#### تحديث صفحة قائمة البرامج
- إضافة أعمدة جديدة: النوع، نطاق المستويات (بشكل ملون)، المجال المستهدف
- فلاتر جديدة: فلترة بالنوع، بالمستوى، بالمجال المستهدف
- Badge ملون يعرض نطاق المستويات

#### عرض خريطة البرامج (Programs Map) — صفحة جديدة
- **`/admin/programs/map`** — عرض بصري لجميع البرامج وعلاقاتها:
  - محور أفقي: المستويات (من 1 إلى N)
  - محور رأسي: أنواع البرامج (training, mobility, therapeutic)
  - كل برنامج يظهر ككارت في المكان المناسب
  - الأسهم تربط البرامج في السلسلة (prerequisite → current → next)
  - يساعد الأدمن على رؤية "هل كل مستوى مغطى ببرنامج؟"

### 3.3 — تحديثات الجلسات والتمارين داخل البرنامج

#### تحديث واجهة تعديل الجلسة (Session Editor)
الحالي يدعم: التمرين، المجموعات، التكرارات، الوقت، الوزن، الراحة.

**إضافة حقول جديدة** (متوافقة مع Schema الحالي):
- **تصنيف الجلسة** (`sessionCategory`): رئيسية | إحماء | تبريد | تكميلية
- **المدة المتوقعة** (`estimatedDurationMin`): بالدقائق

#### تحديث واجهة تعديل عنصر الجلسة (Session Item Editor)
**إضافة حقول جديدة**:
- **مستوى الصعوبة المخصص** (`difficultyCode`): سهل | متوسط | صعب | مخصص
- **نطاق الحركة المستهدف** (`targetROM`): نسبة مئوية (50-100%)
- **درجة الأداء المستهدفة** (`targetFormScore`): نسبة مئوية (50-100%)
- **تجاوز المستوى** (`levelOverride`): رقم اختياري — لتخصيص مستوى مختلف لهذا التمرين
- **بدائل** (`alternatives`): قائمة تمارين بديلة (multi-select من التمارين)

## المرحلة 4: إدارة قواعد التقدم (Progression Rules)

### 4.1 — الهدف
تمكين الأدمن من إدارة قواعد الـ Progression Engine من الداشبورد بدلاً من الاعتماد على seed data فقط.

### 4.2 — التغييرات المطلوبة

#### Backend
- **إنشاء `ProgressionRulesAdminController`**:
  - `GET /admin/progression-rules` — قائمة القواعد
  - `POST /admin/progression-rules` — إنشاء قاعدة
  - `PUT /admin/progression-rules/:id` — تعديل قاعدة
  - `DELETE /admin/progression-rules/:id` — حذف قاعدة
  - `PUT /admin/progression-rules/:id/toggle` — تفعيل/تعطيل

#### Admin Dashboard
- **صفحة إدارة قواعد التقدم** (`/admin/progression-rules`):
  - جدول يعرض: الاسم، النطاق (global/program/exercise)، المحفز، الأولوية، الحالة (مفعل/معطل)، عدد مرات التنفيذ
  
- **نموذج إنشاء/تعديل قاعدة**:
  - **الاسم** (عربي + إنجليزي)
  - **النطاق** (`scope`): عام | برنامج محدد | تمرين محدد
  - **البرنامج** (إذا scope = program): dropdown
  - **التمرين** (إذا scope = exercise): dropdown
  - **المحفز** (`trigger`): اكتمال جلسة | اكتمال أسبوع | اكتمال مجموعة
  - **الأولوية** (`priority`): slider رقمي (1-100)
  - **الشروط** (`conditions`): Rule builder بصري:
    - اختيار المقياس: avgFormScore | completionRate | avgROM | totalVolume | symmetryScore
    - اختيار العامل: >= | <= | > | < | ==
    - القيمة (رقم)
    - النافذة الزمنية: آخر جلسة | آخر جلستين | آخر أسبوع | كامل البرنامج
    - زر "+" لإضافة شرط إضافي (AND)
  - **الإجراء** (`action`):
    - النوع: زيادة الوزن | تخفيض الوزن | زيادة التكرارات | تخفيض التكرارات | زيادة المجموعات | اقتراح إعادة تقييم
    - المقدار (رقم)
    - رسالة الإشعار (عربي + إنجليزي)

- **عرض إحصائيات كل قاعدة**: عدد مرات التنفيذ، آخر تنفيذ، متوسط التأثير — مرتبط بـ `/admin/analytics/rules`

## المرحلة 5: لوحة معلومات شاملة (Analytics Dashboard)

### 5.1 — الهدف
ربط كل الأنظمة السابقة بلوحة معلومات تعطي الأدمن رؤية واضحة عن فعالية النظام.

### 5.2 — الصفحات المطلوبة

#### الصفحة الرئيسية (`/admin`)
تحديث الصفحة الرئيسية الحالية لتعرض:
- إجمالي المستخدمين النشطين
- توزيع المستخدمين على المستويات (bar chart بألوان المستويات)
- عدد الاختبارات هذا الأسبوع/الشهر
- أكثر البرامج تسجيلاً
- معدل إكمال البرامج

#### صفحة تحليل البرامج (`/admin/analytics/programs`)
- فعالية كل برنامج: معدل الإكمال، متوسط التحسن في Body Score
- البرامج الأكثر وصفًا من الـ Prescription Engine
- البرامج التي يغادرها المستخدمون مبكرًا

#### صفحة تحليل المستويات (`/admin/analytics/levels`)
- توزيع المستخدمين على المستويات بمرور الوقت
- متوسط الوقت للترقي من مستوى لآخر
- المستويات التي يتعثر فيها المستخدمون

#### صفحة تحليل الاختبارات (`/admin/analytics/assessments`)
- عدد الاختبارات لكل قالب
- متوسط الـ Body Score لكل قالب
- التمارين الأكثر فشلاً (تحتاج مراجعة)
- معدل التحسن بين الاختبار الأول وإعادة التقييم

## المبادئ التوجيهية

### ما لا يتغير (حسب القرار)
| القرار | السبب |
|--------|-------|
| Progression Engine يبقى تلقائي | المستخدم يقدر يعدل الوزن قبل التمرين + يقدر يعدل Session في البرنامج |
| لا Exercise-level edits من الـ UI | التعديلات الجزئية الحالية كافية |
| المستوى يتغير فقط من Assessment | هذا هو المنطق الطبيعي — التقييم هو المقياس الحقيقي |

### مبادئ المرونة
1. **كل شيء يُقرأ من DB** — لا ثوابت في الكود لبيانات قابلة للتغيير
2. **Cache عند الضرورة** — المستويات والقوالب تتغير نادرًا، فيمكن تخزينها مؤقتًا
3. **Offline-first على Android** — كل بيانات التهيئة تُخزن محليًا
4. **حماية التكامل** — لا يمكن حذف عنصر مرتبط بعناصر أخرى بدون تأكيد
5. **توثيق ذاتي** — كل عنصر في الداشبورد يتضمن tooltips توضيحية
6. **ثنائي اللغة** — كل نص يظهر للمتدرب يجب أن يكون عربي + إنجليزي
